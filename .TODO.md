# TODO: Modularize `canvas.cpp`

## Goal

Reduce the size and mixed responsibilities of `src/canvas.cpp` without over-engineering the project. The current file handles persistence, input, viewport controls, layer operations, undo/redo, drawing tools, UI rendering, and export logic in one place.

Prefer gradual extraction. Keep `Canvas` as the main owner at first, and split along existing responsibility boundaries.

## Recommended Order

1. Split `Canvas::Update()` into smaller private methods inside `canvas.cpp`.
2. Move save/load/export logic into a persistence module.
3. Split rendering into focused render helpers or a separate `CanvasRender.cpp`.
4. Move `Layer` out of `canvas.h` into its own files.
5. Add small helpers for history and palette management once the simpler splits are stable.

## 1. Break Up `Canvas::Update()`

`Canvas::Update()` is currently the largest pressure point. It mixes input collection, command routing, viewport manipulation, undo/redo, layer commands, color picking, and drawing.

Start by keeping everything as private `Canvas` methods:

```cpp
void Canvas::Update() {
    PointerInput input = read_pointer_input();
    update_color_picker_layout();

    if (handle_viewport_shortcuts(input)) return;
    if (handle_layer_shortcuts(input)) return;
    if (handle_color_picking(input)) return;

    handle_tool_input(input);
    prevMousePos = input.pos;
}
```

Suggested private method groups:

- `read_pointer_input()`
- `update_color_picker_layout()`
- `handle_pan_input(...)`
- `handle_zoom_input(...)`
- `handle_rotation_input(...)`
- `handle_layer_shortcuts(...)`
- `handle_tool_shortcuts(...)`
- `handle_color_picking(...)`
- `handle_drawing_input(...)`

Avoid introducing a large input framework for now. The first win is making each branch readable and isolated.

## 2. Extract Persistence

Move file loading, saving, and PNG export out of `canvas.cpp`.

Current responsibilities to extract:

- constructor load logic
- `Canvas::save()`
- `Canvas::save_to_png()`

Suggested files:

- `include/CanvasFile.h`
- `src/CanvasFile.cpp`

Possible API shape:

```cpp
struct CanvasFileData {
    int width;
    int height;
    std::deque<Color> colors;
    std::deque<Layer> layers;
};

bool LoadCanvasFile(const std::string& path, CanvasFileData& out);
bool SaveCanvasFile(const std::string& path, const CanvasFileData& data);
bool ExportCanvasPng(const std::string& path, const std::deque<Layer>& layers, int width, int height);
```

A simpler first version can keep `Canvas::save()` and `Canvas::save_to_png()` as thin wrappers that call helper functions.

## 3. Split Rendering

`Canvas::Render()` currently draws the canvas layers, palette swatches, layer list text, tool state text, color picker, and cursor preview.

Suggested split:

- `render_layers()`
- `render_palette()`
- `render_layer_overlay()`
- `render_tool_overlay()`
- `render_color_picker()`
- `render_cursor_preview()`

Possible file:

- `src/CanvasRender.cpp`

These can remain private `Canvas` methods. A separate renderer class is probably unnecessary unless rendering grows much more.

## 4. Move `Layer` Into Its Own Module

`Layer` currently lives inline in `include/canvas.h` and performs GPU resource management in the header.

Suggested files:

- `include/Layer.h`
- `src/Layer.cpp`

Useful methods to consider:

```cpp
class Layer {
public:
    Layer(int width, int height, bool whiteBackground = false);
    ~Layer();

    Image Snapshot() const;
    void UpdateFromPixels(const void* pixels);
    void Clear(bool whiteBackground = false);
};
```

This would make `canvas.h` smaller and reduce rebuild pressure from implementation details.

## 5. Group Related Canvas State

Before creating manager classes, group related fields into small structs. This makes ownership clearer while keeping the design simple.

Possible structs:

```cpp
struct ViewportState {
    Vector2 canvasPos;
    float scale;
    float rotation;
    bool isMirror;
};

struct ToolState {
    bool isBrush;
    float brushSize;
    float eraserSize;
    float pressure;
    Color color;
    Color previewColor;
    unsigned char transparency;
};

struct PointerState {
    Vector2 pos;
    Vector2 prevPos;
    bool pressed;
    bool down;
    bool released;
    bool penDown;
    bool penInProximity;
};
```

These structs can live privately near `Canvas` first. Move them to headers only when another module needs them.

## 6. Extract Undo/Redo History

Undo and redo currently manage `Image` ownership inline. That is risky because `Image` memory must be unloaded correctly.

Suggested files, if the logic grows:

- `include/History.h`
- `src/History.cpp`

Possible API:

```cpp
class History {
public:
    void PushSnapshot(size_t layerIndex, Image image);
    bool Undo(std::deque<Layer>& layers);
    bool Redo(std::deque<Layer>& layers);
    void ClearRedo();
};
```

The main value is centralizing `UnloadImage()` responsibility and enforcing a maximum undo depth in one place.

## 7. Extract Palette Logic

Recent color management is currently mixed into input and rendering.

Suggested helper:

```cpp
class Palette {
public:
    void AddRecent(Color color);
    bool Contains(Color color) const;
    Color Get(size_t index) const;
    size_t Size() const;
};
```

This would replace the free `contains(...)` helper and keep color queue behavior isolated.

## Avoid For Now

Avoid adding these until the smaller splits prove they are needed:

- `ToolManager`
- `InputSystem`
- command bus architecture
- inheritance-heavy tool classes
- large renderer abstraction

The project is still small. The goal is clearer boundaries, not a framework.

## Definition of Done

A good first modularization pass should leave the project with roughly this shape:

```text
include/
  canvas.h
  Layer.h
  CanvasFile.h

src/
  canvas.cpp
  CanvasRender.cpp
  CanvasFile.cpp
  Layer.cpp
  SDLHandler.cpp
  main.cpp
```

After the first pass, `canvas.cpp` should mostly coordinate behavior instead of containing every implementation detail directly.
